# 協作指南

本專案採兩角色分工，各自的介面不同：**消防專業人員只接觸文件（HTML／紙本），不需操作 GitHub 或 AI**；所有 git、AI、JSON 操作由架構管理者執行。

## 角色與職責

| 角色 | 職責 | 工作介面 |
|------|------|---------|
| **架構管理者** | 架構開發、流程管控、規則編碼（先紅再綠）、核定表匯出與回填、git/PR/CI | 本 repo 全部 |
| **消防專業人員** | 法條審查（核定規則參數）、法規解釋說明（函釋、但書、實務認定） | 核定表 HTML／紙本、審查產出文件 |

## 消防專業人員的三種協作方式（皆不需碰系統）

1. **規則核定**：收到核定表（瀏覽器開啟或紙本），逐條對照法條清單勾選 正確／錯誤＋更正值，簽名回傳。流程詳見 `governance/README.md`
2. **法規解釋**：口述或寫 memo（函釋文號、但書適用、實務認定慣例），由架構管理者轉錄進規則的 `note` 欄位或 `skills/regulation-data.md`
3. **誤判回饋**：審查產出（問題清單、打勾檢核 HTML）本來就是給人讀的——直接指出「這條判錯了」＋理由即可，由架構管理者轉成測試案例修正規則

## 架構管理者的變更規範

### 改 `rules/`（規則庫）

- **必須走先紅再綠**（完整紀律見 `skills/red-green.md`，改編自 obra/superpowers 的 TDD skill）：先在 `rule_tests.json` 寫測試（expected 逐字抄錄法條 PDF、附頁碼與 quote）→ `run-tests --verify-red {測試ID}` 確認紅得正確 → 編碼最小參數 → `run-tests --strict` 轉綠
- **鐵律**：發現參數先於測試被寫入，刪除該參數重來，不保留當參考
- `verified: true` 只能經 `tools/verification_sheet.py apply` 產生，且 results JSON 必附簽名存證路徑
- PR 說明必附法條依據（條號＋頁碼）

### 改 `skills/`（工作流）

- 說明改動了哪個步驟、為什麼；涉及關卡（人工確認、抽檢准出、先紅再綠）的刪改需在 PR 中特別標注

### 改 `tools/`

- 維持 stdlib-only（`pdf_annotate.py` 的 pymupdf 為唯一例外）
- 改動計算邏輯必須有對應測試案例

## 流程規範

1. 所有變更走分支 ＋ PR，不直接 push main
2. CI（`.github/workflows/rule-tests.yml`）自動跑 `self-test` 與 `run-tests --strict`，紅燈不得合併
3. commit message 用中文，說清楚改了什麼、法源依據是什麼
4. `input/` 內的圖面與法規 PDF 若涉及實際案件，先確認可否入庫（個資／機密），不確定就不要 commit

## 品質底線（不可協商）

- 每項判定附法條條號；引不到條號的降級為建議事項
- 禁止心算、禁止憑記憶引法規數值
- 不確定就標「需人工判讀」，不用推測填充
- 本專案輸出僅供審圖輔助，最終判斷歸屬專業消防人員

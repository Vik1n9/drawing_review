# 審圖團隊：四類設備並行審查框架

對 $ARGUMENTS（案件名）進行團隊化消防審圖。依《各類場所消防安全設備設置標準》的四大設備分類，啟動 4 個獨立審查 Agent 並行作業，Team Lead 彙整審查意見書。適用於大型／複雜案件的完整審查；單純案件用 `/code-requirements` + `/gap-analysis` 即可。

## 前置條件

`output/{案件名}-{YYYYMMDD}/case.json` 已完成人工確認（`/plan-intake` 關卡1通過）。未完成則先執行 `/plan-intake`。

## 執行流程

### 第一步：展示團隊框架

向使用者展示以下團隊結構，確認後啟動：

| 角色 | 審查範圍 | 對應法規章節 |
|------|---------|------------|
| **team-lead**（你自己） | 統籌、彙整、意見書定稿 | 全部 |
| **滅火設備審查員** | 滅火器、室內／室外消防栓、自動撒水、水霧／泡沫／乾粉等 | 滅火設備章 |
| **警報設備審查員** | 火警自動警報、手動報警、緊急廣播、瓦斯漏氣火警自動警報 | 警報設備章 |
| **避難逃生設備審查員** | 標示設備（出口標示燈、避難方向指示燈）、避難器具、緊急照明 | 避難逃生設備章 |
| **搶救必要設備審查員** | 連結送水管、消防專用蓄水池、排煙設備、緊急電源插座、無線電通信輔助設備 | 消防搶救上之必要設備章 |

### 第一步半：圖說可讀性與規則庫狀態評估

啟動前向使用者展示兩項風險評估，並告知每個 Agent：

1. **圖說可讀性評級**（承 `/plan-intake` 的 A/B/C）：C 級案件各 Agent 轉為「清單引導模式」，只列應檢核項目與所需資料，不做數量判定
2. **規則庫核定率**：`rules/equipment_rules.json` 中本案涉及規則的 `verified: true` 比例；未核定規則的判定結果一律附警語

**關鍵提醒**：圖面清晰 ≠ 判定可靠。每個 Agent 對視覺判讀不確定的項目必須標「需人工判讀」，這是安全底線不是加分項。

### 第二步：建立團隊與任務

- TeamCreate：team_name `{案件名}-review`（英文小寫）
- TaskCreate 建立 4 個任務，每個任務的 description 包含：
  1. 該類別下每一設備的門檻判斷（**必須用 Bash 呼叫 `tools/fire_code_calc.py check-threshold`**，禁止心算）
  2. 應設設備的數量計算（呼叫對應子命令：extinguisher / hydrant-coverage / sprinkler / detector / occupancy）
  3. 與 case.json 既有設備比對，產出該類別缺失清單（四級：重大缺失／一般缺失／配置疑義／需人工判讀）
  4. 免設判定同樣附計算過程與條號
  5. 該類別的「AI 最可能漏判點」反向自檢

### 第三步：並行啟動 4 個 Agent（同一條訊息）

每個 Agent 配置：`subagent_type: general-purpose`、`run_in_background: true`、`team_name` 對應團隊、`name` 對應角色。

Prompt 模板：

```
你是「{案件名}」消防審圖團隊中的「{角色名}」，負責{設備類別}的法定需求審查。

請完成任務 #{任務編號}：{任務subject}

正典資料：output/{案件名}-{YYYYMMDD}/case.json（已人工確認，以此為準，不要重新解讀圖片）
規則庫：rules/equipment_rules.json
法規版本：{regulation_version}
圖說可讀性評級：{A|B|C}

具體要求：
{任務description}

**審查方法（強制）**：
- 所有門檻判斷與數量計算必須透過 Bash 呼叫 python3 tools/fire_code_calc.py，工具輸出原文嵌入報告
- 每項判定附法條條號；引不到條號的降級為「建議事項」
- 規則庫 verified: false 的參數，保留工具警語
- 不確定就標「需人工判讀」，禁止推測填充

**輸出要求**：
- 逐層檢核表：檢核項｜法條｜應設要求｜圖面現況｜判定｜說明
- 該類別缺失清單（四級分類）
- 末尾附「本類別 AI 審查局限性」3 條以內

**完成後**：
1. TaskUpdate 將任務 #{任務編號} 標記 completed
2. SendMessage 把完整審查報告發給 team-lead（type: "message", recipient: "team-lead"）
```

### 第四步：進度追蹤與接收

- 即時向使用者展示進度表（四類設備審查狀態）
- 每收到一份報告，展示該類別缺失統計（重大 N／一般 N／疑義 N／需人工判讀 N）
- 等待 4 份全部到齊

### 第五步：關閉團隊成員

全部收齊後對 4 個 Agent 發送 shutdown_request（SendMessage, type: "shutdown_request"）。

### 第六步：彙整審查意見書

固定結構：

1. **案件摘要**：基本資料、法規版本、圖說可讀性評級、規則庫核定率
2. **審查結論總表**：四類設備各一列（缺失統計＋最嚴重等級）
3. **全案缺失清單**：四類合併，按 重大→一般→疑義→需人工判讀 排序，逐項附法條
4. **缺失設備數量彙總表**：設備｜樓層｜應設｜既有｜缺口（給補件方直接使用）
5. **需人工判讀與建議補件清單**
6. **免設項目與依據**（供覆核）
7. **AI 審圖局限性聲明**：本次分析信心度低的環節、未核定規則參數清單
8. **計算記錄附錄**：各 Agent 的工具輸出原文
9. **免責聲明**

### 第七步：抽檢准出（關卡）

1. `python3 tools/fire_code_calc.py run-tests --strict` 必須全綠（先紅再綠）
2. 從全案檢核項隨機抽 15%（至少 5 項），team-lead 親自用 `fire_code_calc.py` 重算，比對各 Agent 結果
3. 逐項驗證缺失清單條號存在於規則庫 `legal_basis`
4. 全部通過 →【准出】；任一不符 →【打回】該 Agent 類別重審

### 第八步：產出交付物與清理

- 依 `/gap-analysis` 第六步的流程，產出三項交付物到 `output/{案件名}-{YYYYMMDD}/`：圖面審查 HTML（`dxf_svg_review.py`）、問題清單、法條檢核清單 HTML（`checklist_html.py`）
- TeamDelete 清理團隊資源

## 重要注意事項

1. **4 個 Agent 必須並行啟動**——同一條訊息中呼叫 4 次 Task 工具
2. **Agent 以 case.json 為準**——禁止各自重新解讀原始圖片，避免四份報告基礎資料不一致
3. **計算全部走工具**——LLM 在此系統中只負責萃取、比對敘述與意見書撰寫
4. **需人工判讀是合法輸出**——寧可留白，也不要讓審查者拿到一份「看起來完整但埋了推測」的意見書
5. 遵守 `CLAUDE.md` 審圖最高原則

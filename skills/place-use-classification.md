# 場所用途與樓層屬性判定

對 $ARGUMENTS（案件名）執行《各類場所消防安全設備設置標準》第 12 條場所用途候選判定，供 `/plan-intake` 建立 `case.json` 的 `use_category` 欄位。這個 skill 只產生「用途候選與人工確認清單」，不得取代消防專業人員的最終用途判定。

## 啟用時機

- 建立或修正 `case.json` 時，需要判定各樓層、空間或整棟建築屬於甲、乙、丙、丁、戊類場所
- 審查文件、平面圖房名、使用執照或申請書出現用途名稱，需要轉成第 12 條的法定分類
- 案件有複合用途、地下層、屋突層、屋頂層、無開口樓層或用途邊界疑義

## 法源與位階

先按需載入條文，不要憑記憶引用：

```bash
python3 tools/regulation_index.py lookup --article '§12'
python3 tools/regulation_index.py lookup --article '§4'
```

引用格式固定精確到「條、款、目」：

```text
§12 第1款第5目（甲類場所：餐廳、飲食店、咖啡廳、茶藝館）
§12 第2款第6目（乙類場所：辦公室、診所等）
§12 第5款第3目（戊類場所：地下建築物）
```

第 12 條的「一、二、三、四、五、六」記為第 1 款至第 6 款；各款下的「（一）、（二）...」記為第 1 目、第 2 目。若條文沒有對應目，例如「其他經中央主管機關公告之場所」，引用 `§12 第6款`。

## 基本原則

1. **AI 只列候選，不定稿**：`use_category` 的最終值必須經使用者或消防專業人員確認後，才可改為 `source: "manual"`、`confidence: "high"`。
2. **用途判定優先看文件**：使用執照、申請書、審查表、用途說明與房間名稱優先於圖面符號；DXF/PDF 只作為證據來源。
3. **複合用途保守處理**：同棟或同層有兩種以上第 12 條用途，且管理及使用形態未明確從屬於主用途者，列為戊類候選並標 `需人工判讀`。
4. **地下層不等於地下建築物**：地下樓層先標 `floor_position: "basement"`；是否屬 `§12 第5款第3目` 的「地下建築物」需依案件文件或專業判定確認。
5. **無開口樓層獨立判定**：一般樓層是否為無開口樓層，依 §4 有效開口條件判斷；資料不足時標 `需人工判讀`，不得因樓層普通或圖面不清就自動判定。
6. **屋突層標為屋頂層**：屋突層、R 樓、屋頂機房等先標 `floor_position: "roof_level"`；有實際使用用途時仍需依 §12 另列用途候選。

## 樓層屬性欄位

在 `case.json` 每一樓層補齊以下欄位，與用途分類分開存放：

```json
{
  "floor": "B1F",
  "floor_position": {
    "value": "basement",
    "label": "地下層",
    "confidence": "high",
    "source": "drawing"
  },
  "floor_condition": {
    "windowless": {
      "value": null,
      "label": "無開口樓層需人工判讀",
      "legal_basis": "§4",
      "confidence": "low",
      "source": "derived"
    }
  }
}
```

樓層位置建議值：

| 值 | 中文 | 判定方式 |
|---|---|---|
| `basement` | 地下層 | 樓層名稱為 B、B1、地下、一層地下等 |
| `ordinary` | 一般樓層 | 1F 以上且非屋突、屋頂層 |
| `roof_level` | 屋頂層／屋突層 | R、RF、屋突、屋頂機房、梯間突出物等 |

## 執行流程

### 第一步：收集用途證據

逐層、逐空間列出用途證據，來源優先序如下：

1. 使用執照、變更使用執照、建築物概要表
2. 消防安全設備審查表、申請書、用途說明書
3. 平面圖房名、面積表、圖例或設備配置圖
4. 使用者口頭補充

每筆證據記錄：

```json
{
  "floor": "1F",
  "space": "客席區",
  "evidence_text": "餐廳",
  "source_document": "input/範例/消防安全設備審查表.pdf",
  "source_type": "審查文件",
  "confidence": "high"
}
```

### 第二步：列用途候選

依證據文字比對第 12 條用途速查，輸出候選清單。完全命中的候選可給 `confidence: "medium"`；涉及「類似場所」、機構限制、是否住宿、是否 H-2、是否附屬用途或是否複合用途者，一律 `confidence: "low"` 並列人工確認問題。

候選格式：

```json
{
  "value": "甲5",
  "category": "甲類場所",
  "label": "餐廳、飲食店、咖啡廳、茶藝館",
  "legal_basis": "§12 第1款第5目",
  "matched_terms": ["餐廳"],
  "evidence": ["1F 房名：餐廳"],
  "confidence": "medium",
  "source": "drawing"
}
```

### 第三步：處理複合用途

同一棟或同一防火區劃內有多種用途時，先列出各用途候選，再判斷是否需列戊類候選：

| 情境 | 候選 |
|---|---|
| 有任一甲類用途，且非明確從屬於單一主用途 | `戊1`，`§12 第5款第1目` |
| 無甲類，但有乙、丙、丁類兩種以上用途，且非明確從屬於單一主用途 | `戊2`，`§12 第5款第2目` |
| 僅為地下樓層，但不是法規或文件稱「地下建築物」 | 只標 `floor_position: basement`，不要直接套 `戊3` |

複合用途是否「從屬於主用途」通常需人工判讀；不得只因房名多樣就自動定稿為戊類。

### 第四步：人工確認關卡

把每一層的用途候選、法條位階、證據來源與疑問列成確認表：

| 樓層 | 空間 | 候選分類 | 法條位階 | 證據 | 信心度 | 需確認事項 |
|---|---|---|---|---|---|---|
| 1F | 客席區 | 甲5 | §12 第1款第5目 | 房名：餐廳 | medium | 是否全層均作餐飲使用 |
| B1 | 停車空間 | 丙3 | §12 第3款第3目 | 房名：停車場 | medium | 是否為依法附設室內停車空間 |

使用者確認後，才寫入 `case.json`：

```json
{
  "use_category": {
    "value": "甲5",
    "category": "甲類場所",
    "label": "餐廳、飲食店、咖啡廳、茶藝館",
    "legal_basis": "§12 第1款第5目",
    "confidence": "high",
    "source": "manual",
    "confirmed_by": "user"
  },
  "use_candidates": [
    {
      "value": "甲5",
      "legal_basis": "§12 第1款第5目",
      "confidence": "medium",
      "source": "drawing"
    }
  ]
}
```

### 第五步：輸出摘要

在 `/plan-intake` 的收件摘要中加入「場所用途判定摘要」：

- 各樓層最終用途分類與法條位階
- 未定稿候選與需人工判讀原因
- 地下層、屋頂層／屋突層、無開口樓層的判定狀態
- 複合用途是否列戊類候選，以及尚缺的確認資料

## 不得做的事

1. 不得把用途候選直接當作最終 `use_category`
2. 不得把地下樓層直接等同 `§12 第5款第3目` 地下建築物
3. 不得把屋突層或屋頂機房視為免分類；有實際用途仍要列候選
4. 不得在沒有 §4 開口資料時自行判定「無開口樓層」
5. 不得用法條速查表之外的記憶補充新用途；遇到不在第 12 條文字內的用途，列為 `其他/需人工判讀`

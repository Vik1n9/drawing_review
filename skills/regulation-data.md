# 法規資料治理規範

本規範適用於所有涉及消防法規條文與參數的作業（類比 ai-berkshire 的 `financial-data.md`）。**規則庫中的每一條參數必須可追溯到條文原文，且經雙源驗證後才能標記 `verified: true`。**

---

## 法規來源優先級

| 優先級 | 來源 | 用途 |
|--------|------|------|
| 1（主） | **全國法規資料庫**（law.moj.gov.tw） | 條文原文、最新修正日期 |
| 2（副） | **內政部消防署網站**（nfa.gov.tw） | 函釋、解釋令、圖例規範 |
| 參考 | 各縣市消防局審查基準、公會技術手冊 | 實務認定慣例（標注為「實務參考，非法源」） |

## 規則結構化規範（rules/*.json）

每條規則必須包含以下欄位：

```json
{
  "id": "extinguisher-threshold",
  "equipment": "滅火器",
  "category": "滅火設備",
  "legal_basis": "§14",
  "regulation": "各類場所消防安全設備設置標準",
  "params": { "...": "門檻與係數" },
  "verified": false,
  "verified_by": null,
  "verified_date": null,
  "source_checked": ["law.moj.gov.tw 條文原文", "消防署函釋"],
  "note": "口語摘要與已知爭議點"
}
```

### verified 旗標生命週期

| 狀態 | 意義 | 輸出時的處理 |
|------|------|------------|
| `verified: false` | 開發示例／AI 整理，未經專業核定 | 工具輸出強制附警語：「本參數未經消防專業人員核定，以現行法規為準」 |
| `verified: true` | 消防專業人員對照條文原文核定，填入 `verified_by`、`verified_date` | 正常輸出，附條號 |

**核定流程（雙源驗證）**：
1. 從全國法規資料庫取條文原文，與規則參數逐字比對
2. 檢查消防署是否有相關函釋改變適用方式
3. 兩源一致 → 核定；有函釋差異 → 在 `note` 記錄函釋文號與影響

## 法規版本管理

- 規則檔頂層必須有 `regulation_version`（法規名稱＋最新修正日期）
- 每次修法：
  1. 比對修正條文，更新受影響規則，`verified` 重置為 `false` 待重新核定
  2. 舊版規則檔封存為 `equipment_rules-{修正日期}.json`（審查中案件依送件時適用版本）
  3. 執行 `python3 tools/fire_code_calc.py self-test`
- 報告標頭一律注明所依據的 `regulation_version`

## 引用紀律

1. **禁止憑記憶引用法規數值**——所有門檻、係數、距離只能來自規則庫；規則庫沒有的，先走結構化流程入庫，不要在報告中直接手寫
2. **條號精確到項款目**——「§19 第 1 項第 4 款」優於「§19」
3. **函釋與條文分開標**——基於函釋的判定注明函釋文號
4. **實務慣例不冒充法源**——縣市審查基準、公會手冊的內容標「實務參考」

## 常見誤用型態（開發時的失敗模式清單）

| 誤用 | 後果 | 防呆機制 |
|------|------|---------|
| 記憶中的舊法條數值 | 修法後門檻錯誤 | 禁止心算＋規則庫版本欄位 |
| 用途分類張冠李戴 | 整層門檻全錯 | 分類由人工確認，AI 只建議 |
| 面積單位／樓層合計口徑混淆（單層 vs 合計） | 門檻判斷方向性錯誤 | 規則 `params` 明確標 `scope: per_floor / total` |
| 把「涵蓋半徑估算」當精確需求 | 數量誤報 | 工具輸出強制標「估算下限」 |
| 免設條件漏看但書 | 漏判缺失 | 免設判定必附完整計算過程供覆核 |
